# Evaluation Methodology

## Overview

This document describes the evaluation framework used to compare the **baseline** (stateless) pipeline against the **memory-aware** agent pipeline. The goal is to quantify the impact of structured memory on response quality, efficiency, and consistency.

## Evaluation Pipeline

```
Synthetic Sessions (Phase 5)
        │
        ├──► Baseline Pipeline (no memory)  ──► Logs
        │
        └──► Memory-Aware Pipeline          ──► Logs
                                                  │
                                          Metric Functions
                                                  │
                                          Evaluation Result
                                                  │
                                    ┌─────────────┼─────────────┐
                                    ▼             ▼             ▼
                              report.md     report.json    per-session
```

### Pipelines

| Pipeline | Description |
|----------|-------------|
| **Baseline** | Stateless — each query is processed independently with no memory context. Equivalent to a standard RAG pipeline. |
| **Memory-Aware** | Full agent loop — retrieves conversational history, semantic memory, procedural patterns, and session summaries before reasoning. |

## Metrics

### 1. Consistency Score

**What it measures:** Whether the agent gives consistent answers to related questions across sessions on the same topic.

**How it works:**
1. Group all responses by topic
2. Compute pairwise cosine similarity of response embeddings
3. Count pairs above the threshold (default: 0.85)
4. Score = consistent pairs / total pairs

**Range:** 0.0 (completely inconsistent) to 1.0 (perfectly consistent)

### 2. Context Reuse Rate

**What it measures:** How often the memory-aware pipeline actually uses stored memory when responding.

**How it works:**
1. For each turn, check if `memory_reads > 0`
2. Rate = turns with memory reads / total turns

**Range:** 0.0 (never uses memory) to 1.0 (always uses memory)

### 3. Token Efficiency

**What it measures:** Whether memory reduces the total tokens needed to achieve equivalent responses.

**How it works:**
1. Sum all tokens used by the baseline pipeline
2. Sum all tokens used by the memory pipeline
3. Ratio = baseline_tokens / memory_tokens

**Interpretation:**
- `> 1.0` → Memory saves tokens (good)
- `= 1.0` → No difference
- `< 1.0` → Memory uses more tokens (expected overhead for better quality)

### 4. Retrieval Precision

**What it measures:** Of the documents retrieved, how many were actually relevant.

**How it works:**
1. For each query with retrieved documents, compute: relevant ∩ retrieved / retrieved
2. Average across all queries

**Range:** 0.0 (no relevant docs retrieved) to 1.0 (all retrieved docs are relevant)

### 5. Latency Impact

**What it measures:** The latency overhead of using memory.

**How it works:**
1. Collect per-turn latency for both pipelines
2. Compute avg, p50, p95 for each
3. Overhead = memory_avg - baseline_avg

**Output:** Dict with `baseline_avg`, `baseline_p50`, `baseline_p95`, `memory_avg`, `memory_p50`, `memory_p95`, `overhead_ms`

## Synthetic Sessions

Sessions are generated by `src/simulation/session_generator.py` and cover regulatory topics:

- PIX fees and limits
- PIX security and fraud
- PIX dispute resolution
- PIX international transfers
- PIX compliance and regulations

Each session simulates a multi-turn conversation with follow-up questions, topic references, and context-dependent queries that test memory capabilities.

## Running the Evaluation

```bash
# Generate sessions (if not already generated)
python scripts/generate_sessions.py

# Run evaluation
python scripts/run_evaluation.py

# With custom options
python scripts/run_evaluation.py --sessions data/evaluation/synthetic_sessions.json --output data/evaluation
```

## Output

The evaluation produces two files in `data/evaluation/`:

| File | Format | Purpose |
|------|--------|---------|
| `report.md` | Markdown | Human-readable report with tables — portfolio-ready |
| `report.json` | JSON | Machine-readable for programmatic analysis or dashboards |

## Limitations

- **Synthetic sessions**: Generated from templates, not real user interactions. Results indicate relative performance, not absolute quality.
- **Simple embeddings**: The default consistency metric uses character-frequency embeddings as a lightweight fallback. For production evaluation, replace with real model embeddings.
- **Placeholder pipelines**: The CLI script uses placeholder pipeline functions. Full integration with the agent graph requires Phase 3 + Phase 4 components connected.
- **No human evaluation**: All metrics are automated. Human evaluation of response quality is recommended as a complement.
